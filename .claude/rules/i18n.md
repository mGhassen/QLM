# Internationalization (i18n)

All user-facing strings go through the i18n system. **Never hardcode user-facing text.**

## Stack

- **Library**: `react-i18next` (wrapped by `@qlm/i18n`)
- **Locale files**: JSON under `packages/i18n/`
- **Wrapper**: use `@qlm/ui/trans` for the `Trans` component, and the `t()` function from `useTranslation()` for strings

## Rules

### Always

- Ôťů Use `t('namespace.key')` for dynamic strings
- Ôťů Use `<Trans i18nKey="namespace.key" />` from `@qlm/ui/trans` for strings with embedded JSX (links, bold, variables)
- Ôťů Add translation keys in all supported locale JSON files at the same time as adding the usage
- Ôťů Use namespaces to group keys logically (e.g. `notebook.list.title`, `notebook.list.createButton`)
- Ôťů Use interpolation for dynamic values: `t('greeting', { name: user.name })` Ôćĺ `Hello, {{name}}`

### Never

- ÔŁî Hardcoded English (or any language) user-facing strings in JSX, toast messages, error messages, tooltips, placeholders, aria-labels, or table headers
- ÔŁî Import `Trans` from `react-i18next` directly ÔÇö **ESLint enforces `@qlm/ui/trans`**
- ÔŁî String concatenation for translations (`t('hello') + ' ' + user.name`) ÔÇö use interpolation instead
- ÔŁî Translation keys built at runtime from unknown data (breaks static extraction and type inference)

## Code examples

### Simple text

```tsx
// ÔŁî Wrong
<h1>Notebooks</h1>
<button>Create new</button>

// Ôťů Right
import { useTranslation } from 'react-i18next';
const { t } = useTranslation();

<h1>{t('notebook.list.title')}</h1>
<button>{t('notebook.list.createButton')}</button>
```

### Interpolation

```tsx
// ÔŁî Wrong
<p>Welcome, {user.name}!</p>

// Ôťů Right
<p>{t('common.welcome', { name: user.name })}</p>
// Locale: "common.welcome": "Welcome, {{name}}!"
```

### Rich text with JSX

```tsx
// ÔŁî Wrong
import { Trans } from 'react-i18next';
<Trans i18nKey="intro">
  Read the <a href="/docs">documentation</a>
</Trans>

// Ôťů Right ÔÇö use the project wrapper
import { Trans } from '@qlm/ui/trans';
<Trans i18nKey="intro">
  Read the <a href="/docs">documentation</a>
</Trans>
```

### Toast / notifications

```tsx
// ÔŁî Wrong
toast.success('Notebook created');
toast.error('Something went wrong');

// Ôťů Right
toast.success(t('notebook.toast.created'));
toast.error(t('errors.generic'));
```

### Aria labels & placeholders

```tsx
// ÔŁî Wrong
<Input placeholder="Search notebooks..." aria-label="Search" />

// Ôťů Right
<Input
  placeholder={t('notebook.list.searchPlaceholder')}
  aria-label={t('notebook.list.searchAriaLabel')}
/>
```

## Exceptions

Strings that are **not** user-facing can remain as plain literals:

- Console logs / dev-only messages
- `data-testid` values and other test-only attributes
- Error codes passed to `DomainException.new({ code })` (the `overrideMessage` is also internal ÔÇö real UI error messages come from localized mappings)
- Technical identifiers (CSS class names, HTML IDs, GraphQL queries)

## Adding a new locale key

1. Add the key + default English string to `packages/i18n/src/locales/en/<namespace>.json`
2. Add translations to other locale files in the same namespace
3. Use `t('namespace.key')` in code ÔÇö TypeScript should pick up the new key if the type definitions regenerate
4. If the key has variables, document them in a comment in the JSON file

## ESLint

The custom rule `@qlm/eslint-config` enforces `@qlm/ui/trans` over `react-i18next/Trans`. **Do not disable this rule.**
