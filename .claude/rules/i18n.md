# Internationalization (i18n)

All user-facing strings go through the i18n system. **Never hardcode user-facing text.**

## Stack

- **Library**: `react-i18next` (wrapped by `@guepard/i18n`)
- **Locale files**: JSON under `packages/i18n/`
- **Wrapper**: use `@guepard/ui/trans` for the `Trans` component, and the `t()` function from `useTranslation()` for strings

## Rules

### Always

- ✅ Use `t('namespace.key')` for dynamic strings
- ✅ Use `<Trans i18nKey="namespace.key" />` from `@guepard/ui/trans` for strings with embedded JSX (links, bold, variables)
- ✅ Add translation keys in all supported locale JSON files at the same time as adding the usage
- ✅ Use namespaces to group keys logically (e.g. `notebook.list.title`, `notebook.list.createButton`)
- ✅ Use interpolation for dynamic values: `t('greeting', { name: user.name })` → `Hello, {{name}}`

### Never

- ❌ Hardcoded English (or any language) user-facing strings in JSX, toast messages, error messages, tooltips, placeholders, aria-labels, or table headers
- ❌ Import `Trans` from `react-i18next` directly — **ESLint enforces `@guepard/ui/trans`**
- ❌ String concatenation for translations (`t('hello') + ' ' + user.name`) — use interpolation instead
- ❌ Translation keys built at runtime from unknown data (breaks static extraction and type inference)

## Code examples

### Simple text

```tsx
// ❌ Wrong
<h1>Notebooks</h1>
<button>Create new</button>

// ✅ Right
import { useTranslation } from 'react-i18next';
const { t } = useTranslation();

<h1>{t('notebook.list.title')}</h1>
<button>{t('notebook.list.createButton')}</button>
```

### Interpolation

```tsx
// ❌ Wrong
<p>Welcome, {user.name}!</p>

// ✅ Right
<p>{t('common.welcome', { name: user.name })}</p>
// Locale: "common.welcome": "Welcome, {{name}}!"
```

### Rich text with JSX

```tsx
// ❌ Wrong
import { Trans } from 'react-i18next';
<Trans i18nKey="intro">
  Read the <a href="/docs">documentation</a>
</Trans>

// ✅ Right — use the project wrapper
import { Trans } from '@guepard/ui/trans';
<Trans i18nKey="intro">
  Read the <a href="/docs">documentation</a>
</Trans>
```

### Toast / notifications

```tsx
// ❌ Wrong
toast.success('Notebook created');
toast.error('Something went wrong');

// ✅ Right
toast.success(t('notebook.toast.created'));
toast.error(t('errors.generic'));
```

### Aria labels & placeholders

```tsx
// ❌ Wrong
<Input placeholder="Search notebooks..." aria-label="Search" />

// ✅ Right
<Input
  placeholder={t('notebook.list.searchPlaceholder')}
  aria-label={t('notebook.list.searchAriaLabel')}
/>
```

## Exceptions

Strings that are **not** user-facing can remain as plain literals:

- Console logs / dev-only messages
- `data-testid` values and other test-only attributes
- Error codes passed to `DomainException.new({ code })` (the `overrideMessage` is also internal — real UI error messages come from localized mappings)
- Technical identifiers (CSS class names, HTML IDs, GraphQL queries)

## Adding a new locale key

1. Add the key + default English string to `packages/i18n/src/locales/en/<namespace>.json`
2. Add translations to other locale files in the same namespace
3. Use `t('namespace.key')` in code — TypeScript should pick up the new key if the type definitions regenerate
4. If the key has variables, document them in a comment in the JSON file

## ESLint

The custom rule `@guepard/eslint-config` enforces `@guepard/ui/trans` over `react-i18next/Trans`. **Do not disable this rule.**
