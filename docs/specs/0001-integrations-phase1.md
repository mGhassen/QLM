This document is the implementation spec for RFC 0001. The RFC establishes the *why* and *shape*; this spec defines the *what* and *how*: resolved open questions, exact data shapes, API contracts, functional flows, technical flows, file-by-file work items, and a verification plan.

Scope is strict. Everything out of phase 1 (dataplane node provisioning, cloud database ingestion, OIDC, more providers, audit log) is deferred to its own RFC and does not appear here.

---

## 1. Resolved open questions (from RFC §13)

| # | Question                                | Resolution for phase 1                                                                                                                                                         |
| - | --------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 1 | Vault capacity and deletion             | Secrets are serialised to a single JSON string (`JSON.stringify({ ... })`) and passed to `ISecretVault.protect()`. A new **optional** method `forget?(ref): Promise<void>` is added to `ISecretVault`; `DeleteIntegrationConnectionService` calls it if present, otherwise drops the row and logs a warning. |
| 2 | Sidebar bucket                          | `ops` bucket, next to the project dashboard. Revisit in phase 2 when dataplane nodes may justify a new "Infrastructure" bucket.                                                |
| 3 | Slug generation                         | Autogenerated from `name` via `slugify(name)` + numeric suffix on `(project_id, slug)` collision. Users cannot set slugs directly in phase 1.                                  |
| 4 | Driver registry location                | New server-only package `packages/integrations-drivers`. `apps/server` is the only consumer.                                                                                  |
| 5 | Permission naming                       | New `integrations.manage` value added to the `public.app_permissions` enum.                                                                                                  |
| 6 | Supabase-vs-HTTP for reads              | **HTTP-only** for phase 1. All reads and writes go through `/integrations/*`. Simpler layering, no secret-leak risk. Revisit after phase 2.                                   |
| 7 | Error taxonomy                          | Closed set: `invalid_credentials | network | permission_denied | unknown`. `errorMessage` carries provider detail for display. Extend in phase 2 if needed.                    |
| 8 | Dataplane node boundary                 | Out of scope for this spec. Phase 2's RFC will decide whether nodes are project-scoped or cross-project.                                                                       |

## 2. User stories

The spec covers exactly these user stories. Anything not in this list is out of phase 1.

| ID   | As a…                 | I want to…                                                                | So that…                                                                                    |
| ---- | --------------------- | ------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| US-1 | project admin         | open the Integrations app in the project sidebar                          | I can see which cloud accounts this project is already connected to                        |
| US-2 | project admin         | create a new AWS integration by pasting access key credentials            | my project can talk to my AWS account                                                        |
| US-3 | project admin         | create a new GCP integration by pasting a service-account JSON             | my project can talk to my GCP project                                                        |
| US-4 | project admin         | test an existing integration's credentials on demand                      | I know the credentials still work before phase-2 flows depend on them                        |
| US-5 | project admin         | see which regions an integration's credentials can reach                  | I can verify the credentials have the scope I expected                                       |
| US-6 | project admin         | rename an integration                                                      | the UI stays meaningful as I accumulate integrations                                         |
| US-7 | project admin         | rotate an integration's credentials                                        | I can react to a rotation in AWS/GCP IAM without recreating the row                          |
| US-8 | project admin         | delete an integration                                                      | I can clean up what I no longer need                                                          |
| US-9 | project member (non-admin) | see the list of integrations and their last-test status                | I have visibility without the ability to mutate them                                          |

#### 3.2.4 Credentials form — AWS

Fields (all i18n'd):

| Field                 | Control              | Required | Validation                                          |
| --------------------- | -------------------- | -------- | --------------------------------------------------- |
| Name                  | text input           | yes      | 1..60 chars, trims whitespace                        |
| AWS access key id     | text input           | yes      | matches `/^AKIA[0-9A-Z]{16}$|^ASIA[0-9A-Z]{16}$/`    |
| AWS secret access key | password input       | yes      | non-empty, 40 chars typical (no strict regex)        |
| Session token         | textarea, collapsed  | no       | non-empty if provided                                |
| Default region        | select (hardcoded list of common regions + "Other…") | yes | one of the known ids |

Two buttons at the bottom:

- **Test connection** (secondary): runs a *dry* test — calls `POST /integrations/test-draft` (see §5.2) with the form values. Does **not** persist. Shows status inline.
- **Save** (primary): `POST /integrations`. Persists the row with the test result if the user tested first; otherwise the row lands with `test_status='untested'`.

#### 3.2.5 Credentials form — GCP

Fields:

| Field                 | Control              | Required | Validation                                          |
| --------------------- | -------------------- | -------- | --------------------------------------------------- |
| Name                  | text input           | yes      | 1..60 chars                                         |
| Service-account JSON  | textarea (monospace, 12 rows) | yes | must parse as JSON with `type=="service_account"`, `project_id`, `client_email`, `private_key` present |
| Default region        | select               | yes      | one of known GCP regions                             |
| Project id            | text, auto-filled from JSON `project_id` once parsed | yes | non-empty |

Same two-button footer (Test + Save).

The textarea is `type="password"` equivalent: the field is rendered monospace but the server response never echoes its content back, so editing after save is only possible via **Rotate credentials**, which clears the textarea before opening.

### 3.3 User flows (happy paths)

#### 3.3.1 First-time integration creation (AWS)

1. Admin opens project → clicks **Integrations** in sidebar.
2. Sees empty state, clicks **New integration**.
3. Picks **AWS** on the provider card.
4. Fills in name (*"prod-aws"*), access key, secret key, region (*us-east-1*).
5. Clicks **Test connection**.
   - Spinner for up to ~5 s.
   - Success → inline green banner *"Connected as arn:aws:iam::123456789012:user/qlm"*. Save button becomes primary-emphasised.
6. Clicks **Save**.
7. Toast *"Integration created"*. Navigates to `/integrations/prod-aws`.

#### 3.3.2 Test a stale integration

1. Admin opens `/integrations/staging`.
2. Detail shows status *"Never tested"*.
3. Clicks **Test**.
4. Status pill flips to spinner → green or red. On red, error message is shown below the pill (`integrations.test.error.<errorCode>`).

#### 3.3.3 Rotate credentials

1. Admin opens `/integrations/prod-aws`.
2. Clicks **Rotate credentials**.
3. Dialog opens with empty credential fields.
4. Admin pastes new key/secret, clicks **Save**.
5. Server: `ISecretVault.protect(newRaw)` → new `secret_ref` → old `secret_ref` is forgotten via `ISecretVault.forget?()` → `test_status` reset to `untested`.
6. Dialog closes, detail view shows new *Last tested: never*. Admin is expected to re-test.

#### 3.3.4 Delete

1. Admin opens detail view.
2. Ellipsis → **Delete**.
3. Confirm dialog with the integration's name typed to confirm (destructive pattern).
4. Submit → `DELETE /integrations/:id`. Navigate back to list. Toast *"Integration deleted"*.

### 3.4 Error and edge-case behaviour

| Situation                                                   | UX                                                                                                                                  |
| ----------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| Non-admin views the list                                    | List and detail render; mutation buttons are disabled with tooltip `integrations.perm.denied`.                                      |
| Non-admin hits a mutation endpoint directly                 | Server returns 403 with code `permission_denied`; UI toast maps to `errors.permissionDenied`.                                       |
| Slug collision on create                                    | Server increments suffix (`prod-aws`, `prod-aws-2`, …) transparently. User sees the final slug in the URL.                          |
| Server receives creds that fail validation                  | 400 with field-level errors (Zod validation shape). Form highlights the offending field.                                            |
| GCP JSON cannot be parsed                                   | Client-side Zod validation catches it before POST; inline error under the textarea.                                                |
| Test succeeds but `listRegions` later fails                 | Detail view shows the success pill for status but a separate error banner for the regions panel. Retry affects only the regions panel. |
| User rotates credentials to an invalid set                  | `updateCredentials` stores the new secret, resets `test_status='untested'`, and the next explicit **Test** will report the failure. Server does **not** auto-test on rotation. |
| Rate limit exceeded on `POST /:id/test`                     | 429 with `Retry-After`. UI shows `integrations.test.error.rateLimited` and disables the button for the header-supplied duration.    |
| User deletes an integration that is referenced by a phase-2+ object | Not a concern in phase 1 — no such references exist yet. Phase 2's RFC is responsible for the FK behaviour.                |
| Vault.reveal() fails mid-request                            | Server returns 500 with code `vault_unavailable`. UI toast `errors.vaultUnavailable`. Row remains untouched.                        |
| Network error between server and AWS/GCP                    | Mapped to `TestResult.errorCode='network'`; UI shows `integrations.test.error.network` and leaves the old test status intact *unless* the call was an explicit test (which always overwrites). |

## 4. Technical flow

### 4.1 Layered sequence — create AWS integration

```
Browser (apps/web)                 Server (apps/server)            Domain (packages/domain)              Driver (packages/integrations-drivers)   Supabase / Vault
      │                                   │                                │                                        │                                  │
      │  POST /integrations {name, aws:{…}}│                                │                                        │                                  │
      │──────────────────────────────────▶│                                │                                        │                                  │
      │                                   │ zValidator('json', schema)      │                                        │                                  │
      │                                   │─────────────────────────────────▶ CreateIntegrationConnectionService     │                                  │
      │                                   │                                │ (repo, vault)                          │                                  │
      │                                   │                                │                                        │                                  │
      │                                   │                                │ slug ← generate(name)                   │                                  │
      │                                   │                                │ secret ← vault.protect(                 │                                  │
      │                                   │                                │   JSON.stringify(creds),                │                                  │
      │                                   │                                │   { keyName: 'integration:aws:<uuid>' })│                                  │
      │                                   │                                │                                        │                                  │ vault: stores blob, returns opaque ref
      │                                   │                                │◀───────────────────────────────────────│                                  │
      │                                   │                                │ repo.create(entity)                     │                                  │
      │                                   │                                │                                        │                                  │ INSERT row with secret_ref
      │                                   │                                │◀───────────────────────────────────────│                                  │
      │                                   │                                │ return IntegrationConnectionOutput      │                                  │
      │                                   │◀───────────────────────────────│                                        │                                  │
      │◀────────────────────── 201, sanitised DTO (no secret) ─────────────│                                        │                                  │
```

Secrets exist in memory server-side only long enough to call `vault.protect()`. They are never logged, never passed back to the browser, never persisted outside the vault.

### 4.2 Layered sequence — test an existing integration

```
Browser                            Server                          Domain                                   Driver                          Supabase / Vault / AWS
    │                                 │                               │                                        │                                   │
    │ POST /integrations/:id/test     │                               │                                        │                                   │
    │────────────────────────────────▶│                               │                                        │                                   │
    │                                 │ rate-limit check              │                                        │                                   │
    │                                 │───────────────────────────────▶ TestIntegrationConnectionService        │                                   │
    │                                 │                               │ (repo, vault, driverRegistry)          │                                   │
    │                                 │                               │ row ← repo.findById(id)                 │                                   │
    │                                 │                               │────────────────────────────────────────│                                   │ SELECT by id
    │                                 │                               │◀───────────────────────────────────────│                                   │
    │                                 │                               │ raw  ← vault.reveal(row.secret_ref)     │                                   │
    │                                 │                               │◀───────────────────────────────────────│                                   │
    │                                 │                               │ creds ← JSON.parse(raw)                 │                                   │
    │                                 │                               │ driver← driverRegistry.resolve(provider)│                                   │
    │                                 │                               │ result← driver.testConnection(creds)    │                                   │
    │                                 │                               │────────────────────────────────────────▶                                   │ AWS STS GetCallerIdentity
    │                                 │                               │                                        │◀──────────────────────────────────│
    │                                 │                               │ repo.updateTestResult(id, result)       │                                   │
    │                                 │                               │────────────────────────────────────────│                                   │ UPDATE row
    │                                 │                               │ return TestResult (+ updated row)       │                                   │
    │                                 │◀──────────────────────────────│                                        │                                   │
    │◀───────────── 200 TestResult ───│                               │                                        │                                   │
```

Rate limit: a sliding-window counter keyed on `${userId}:${integrationId}` stored in memory on the server. Phase 1 accepts an in-process limiter; phase 2 may promote it to Redis if the server is horizontally scaled.

### 4.3 Layered sequence — list regions

Same shape as 4.2 but `driver.listRegions(creds)` instead of `testConnection`, and no repo update. Regions are never persisted.

### 4.4 Rotate credentials sequence

```
Browser → PUT /integrations/:id/credentials { aws:{…} }
Server → UpdateIntegrationCredentialsService
  ├─ row    ← repo.findById(id)
  ├─ newRef ← vault.protect(JSON.stringify(newCreds), { keyName: 'integration:<provider>:<uuid>' })
  ├─ repo.updateCredentialsRef(id, newRef, { test_status: 'untested', test_identity: null, test_error: null, tested_at: null })
  ├─ vault.forget?.(row.secret_ref)         ← best-effort; log & continue on failure
  └─ return sanitised DTO
```

### 4.5 Delete sequence

```
Browser → DELETE /integrations/:id
Server → DeleteIntegrationConnectionService
  ├─ row ← repo.findById(id)
  ├─ repo.delete(id)
  ├─ vault.forget?.(row.secret_ref)         ← best-effort
  └─ 204
```

### 4.6 Component split — `packages/features/integrations` vs `packages/apps/integrations`

Two packages, following the existing convention set by `packages/features/notebook` + `packages/apps/notebook`:

- **`packages/features/integrations`** — **presentational only**. Every component accepts data and callbacks via props. Zero imports from `@qlm/shell-runtime`, zero React Query, zero repository access, zero network calls. Every component has a `.stories.tsx` file covering every meaningful state (empty, loading, success, error, disabled-by-permission). This is what gets validated in Storybook before any backend work lands.
- **`packages/apps/integrations`** — **thin plugin shell**. `manifest.ts` + `plugin-root.tsx` + pages. Pages import the feature components and wire them to `useShell().integrations` via React Query. No JSX logic of its own beyond glue.

This split is mandatory — not optional — because it is what lets the UI be validated and iterated independently of the domain/server work, which is the whole point of step 1 in §12.

#### 4.6.2 App plugin tree (`packages/apps/integrations`)

```
src/
├─ manifest.ts
```

The app package imports from `@qlm/features-integrations` and adds only the data layer: React Query hooks, mutation handlers, cache invalidation, toast side-effects. If you find yourself writing JSX in `apps/integrations` that isn't one of (a) `useShell()` call, (b) React Query hook, (c) passing props/callbacks into a feature component, it belongs in the feature package instead.

#### 4.6.3 Storybook fixtures

Co-located under `packages/features/integrations/src/__fixtures__/`:

## 5. API contracts

All routes mounted under `/api/projects/:projectId/integrations` (path-parameter style matches existing server routes). Every request is authenticated by the existing session middleware and authorised by the RLS on `integration_connections`.

### 5.1 Data shapes

```ts
// Sanitised DTO — the only shape the client ever sees on the integration itself
type IntegrationConnectionDto = {
  id: string;
  projectId: string;
  provider: 'aws' | 'gcp';
  name: string;
  slug: string;
  config: {
    defaultRegion: string;
    accountHint?: string;   // AWS account id or GCP project id
  };
  testStatus: 'untested' | 'success' | 'failed';
  testIdentity: string | null;
  testError: string | null;
  testedAt: string | null;  // ISO8601
  createdAt: string;
  updatedAt: string;
  createdBy: string | null;
};

type AwsCredentialsInput = {
  provider: 'aws';
  accessKeyId: string;
  secretAccessKey: string;
  sessionToken?: string;
  defaultRegion: string;
};

type GcpCredentialsInput = {
  provider: 'gcp';
  serviceAccountJson: string;   // raw JSON text
  defaultRegion: string;
  // projectId is derived server-side from the parsed JSON
};

type CredentialsInput = AwsCredentialsInput | GcpCredentialsInput;

type TestResult = {
  ok: boolean;
  identity?: string;
  errorCode?: 'invalid_credentials' | 'network' | 'permission_denied' | 'unknown';
  errorMessage?: string;
};

type Region = { id: string; name: string };
```

### 5.2 Endpoints

| Method | Path                                         | Purpose                                                        | Request body                                         | Response                               |
| ------ | -------------------------------------------- | -------------------------------------------------------------- | ---------------------------------------------------- | -------------------------------------- |
| GET    | `/integrations`                              | List integrations for the current project                      | —                                                    | `IntegrationConnectionDto[]`           |
| POST   | `/integrations`                              | Create an integration (secrets in body)                         | `{ name: string; credentials: CredentialsInput }`    | `201 IntegrationConnectionDto`         |
| POST   | `/integrations/test-draft`                   | Dry-test a credentials payload without persisting (used by the create form's "Test connection" button) | `{ credentials: CredentialsInput }` | `200 TestResult`                       |
| GET    | `/integrations/:id`                          | Read one                                                        | —                                                    | `IntegrationConnectionDto`             |
| PATCH  | `/integrations/:id`                          | Update non-secret fields (`name` only in phase 1)               | `{ name?: string }`                                  | `IntegrationConnectionDto`             |
| PUT    | `/integrations/:id/credentials`              | Rotate credentials; resets `testStatus` to `untested`           | `{ credentials: CredentialsInput }`                  | `IntegrationConnectionDto`             |
| DELETE | `/integrations/:id`                          | Delete                                                          | —                                                    | `204`                                  |
| POST   | `/integrations/:id/test`                     | Test an existing integration; updates `testStatus` on the row   | —                                                    | `200 TestResult` (and updated DTO side-returned on the header or refetched) |
| GET    | `/integrations/:id/regions`                  | Live region list; not persisted                                 | —                                                    | `200 Region[]`                         |

Error responses follow the existing server convention:

```json
{ "error": { "code": "permission_denied", "message": "You do not have integrations.manage on this organisation." } }
```

Codes used in phase 1: `validation_error`, `permission_denied`, `not_found`, `slug_conflict` (handled server-side by retry, so rarely surfaced), `rate_limited`, `vault_unavailable`, `provider_error`.

### 5.3 Rate limiting

Only `POST /integrations/:id/test` and `POST /integrations/test-draft` are rate-limited in phase 1.

- Window: 60 seconds, sliding.
- Limit: 10 requests per (user, project).
- Response on limit: `429` with `Retry-After` seconds.
- Implementation: in-memory sliding window inside `apps/server`. Good enough for one server instance; phase 2 will revisit for horizontal scaling.

### 6.1 Schema

```sql
-- Enum extension first — separate transaction per Postgres requirements.
alter type public.app_permissions add value if not exists 'integrations.manage';
commit;

create table if not exists public.integration_connections (
  id              uuid primary key default extensions.uuid_generate_v4(),
  project_id      uuid not null references public.projects(id) on delete cascade,
  provider        text not null check (provider in ('aws', 'gcp')),
  name            text not null check (char_length(name) between 1 and 60),
  slug            text not null check (char_length(slug) between 1 and 80),
  config          jsonb not null default '{}'::jsonb,
  secret_ref      text,
  test_status     text not null default 'untested'
                    check (test_status in ('untested','success','failed')),
  test_identity   text,
  test_error      text,
  tested_at       timestamptz,
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now(),
  created_by      uuid references auth.users(id) on delete set null,
  updated_by      uuid references auth.users(id) on delete set null,
  unique (project_id, slug)
);

create index if not exists ix_integration_connections_project
  on public.integration_connections (project_id);

alter table public.integration_connections enable row level security;
revoke all on public.integration_connections from authenticated, service_role;
grant select, insert, update, delete on public.integration_connections to authenticated;

-- Read: any member of the project's organisation.
create policy "integration_connections_read" on public.integration_connections for select
  to authenticated using (
    exists (
      select 1 from public.projects p
      where p.id = integration_connections.project_id
        and public.has_role_on_organization(p.organization_id)
    )
  );

-- Insert: integrations.manage or org owner, and created_by must match the caller.
create policy "integration_connections_insert" on public.integration_connections for insert
  to authenticated with check (
    created_by = auth.uid() and exists (
      select 1 from public.projects p
      where p.id = integration_connections.project_id
        and (
          public.is_organization_owner(p.organization_id) or
          public.has_permission(auth.uid(), p.organization_id, 'integrations.manage'::app_permissions)
        )
    )
  );

-- Update: same permission gate, both USING and WITH CHECK.
create policy "integration_connections_update" on public.integration_connections for update
  to authenticated using (
    exists (
      select 1 from public.projects p
      where p.id = integration_connections.project_id
        and (
          public.is_organization_owner(p.organization_id) or
          public.has_permission(auth.uid(), p.organization_id, 'integrations.manage'::app_permissions)
        )
    )
  ) with check (
    exists (
      select 1 from public.projects p
      where p.id = integration_connections.project_id
        and (
          public.is_organization_owner(p.organization_id) or
          public.has_permission(auth.uid(), p.organization_id, 'integrations.manage'::app_permissions)
        )
    )
  );

-- Delete: same.
create policy "integration_connections_delete" on public.integration_connections for delete
  to authenticated using (
    exists (
      select 1 from public.projects p
      where p.id = integration_connections.project_id
        and (
          public.is_organization_owner(p.organization_id) or
          public.has_permission(auth.uid(), p.organization_id, 'integrations.manage'::app_permissions)
        )
    )
  );

-- Timestamps & user tracking (project-wide helpers per apps/web/supabase/CLAUDE.md).
create trigger set_integration_connections_timestamps
  before insert or update on public.integration_connections
  for each row execute function public.trigger_set_timestamps();

create trigger set_integration_connections_user_tracking
  before insert or update on public.integration_connections
  for each row execute function public.trigger_set_user_tracking();
```

### 6.2 Config contract

The `config` jsonb column is **typed** in TypeScript but unconstrained in SQL (so future non-breaking extensions are cheap). Phase 1 shape:

```ts
type IntegrationConnectionConfig = {
  defaultRegion: string;
  accountHint?: string;   // AWS account id (12 digits) or GCP project id
};
```

Anything else goes in a new column in a future migration, not in `config`.

### 6.3 Secret payload contract

Vault input is always a JSON string. Phase 1 shapes:

```ts
// AWS
{ accessKeyId: string; secretAccessKey: string; sessionToken?: string }

// GCP (raw JSON text from the service account download — stored as-is inside the outer JSON envelope)
{ serviceAccountJson: string }
```

Not included in the secret blob: `defaultRegion`, `projectId`, `accountHint`. Those are non-secret metadata and live in `config` on the row. This split is deliberate: the day OIDC lands, the secret blob is replaced with a short-lived credential envelope without touching the non-secret side.

## 9. Security checklist

- [ ] Secrets never appear in `config`.
- [ ] DTOs never serialise `secret_ref`.
- [ ] Logging middleware redacts `accessKeyId`, `secretAccessKey`, `sessionToken`, `serviceAccountJson`, `credentials`.

- [ ] Server outbound allowlist covers `sts.amazonaws.com`, `ec2.*.amazonaws.com`, `oauth2.googleapis.com`, `cloudresourcemanager.googleapis.com`, `compute.googleapis.com` (ops task, out-of-band).

```json
      "deleted": "Integration deleted"
    },
    "perm": {
      "denied": "You don't have the integrations.manage permission on this organisation."
    }
  }
}
```

Every key appears in all locale files. Translations can land empty-string in non-English locales to begin with; the i18n fallback layer will show English.

## 12. Implementation sequencing

UI-first. The goal is that the user can visually validate every screen in Storybook **before** a single line of domain/server/adapter code lands. That means steps 1–3 produce no runtime behaviour — just types, components, and stories — and step 1 has a hard review gate.

Each bullet is an atomic commit-sized unit. `pnpm typecheck` runs green after every one.

### Stage A — types and UI (no backend)

1. **Domain types only** — land the *type-level* bits of §7.1 and §7.12 that the UI package depends on:
   - `packages/domain/src/entities/integration-connection.type.ts` (Zod schemas + inferred types)
   - `packages/domain/src/usecases/dto/integration-usecase-dto.ts` (`IntegrationConnectionOutput`, `CredentialsInput`, `TestResult`, `Region`)
   - Barrel exports from `@qlm/domain`.
   - **Not yet**: repository port, services, `Repositories` interface changes, `ISecretVault.forget?`. Those land in stage B.
   - Verification: `pnpm --filter @qlm/domain build` passes, the new types are importable from a dummy file in the feature package (next step).
2. **`packages/features/integrations` scaffolding** — create the package, copy `tsconfig`/`eslint`/`package.json` from `packages/features/notebook`, land `src/index.ts`, the empty `src/integrations-ui.tsx`, and `src/__fixtures__/*.fixture.ts` typed against the stage-1 DTOs. Storybook discovers the package.
3. **Components + stories** — implement every component from §4.6.1 and every story from §4.6.4. Interaction tests for forms (`AwsCredentialsForm`, `GcpCredentialsForm`) land here too (`@testing-library/react` against the pure components — no mocks needed). At the end of this step, Storybook shows the full phase-1 UI driven by fixtures.

   **Review gate — do not proceed past stage A until this is cleared.** The user opens Storybook, walks every story in the §4.6.4 matrix, and either approves the visual/UX or requests changes. Changes are iterated in-place inside stage A. Stage B starts only after approval.

### Stage B — data & domain

4. **Domain: ports + services** — the rest of §7.1: repository port, service classes, `Repositories` interface update, `ISecretVault.forget?` addition. Unit tests for every service.
5. **SQL schema + enum** — §7 work items for `apps/web/supabase/schemas/*`. Run `pnpm supabase:web:reset && pnpm supabase:web:typegen`.
6. **Supabase adapter** (`packages/repositories/supabase/src/integration-connection.repository.ts`). Wire into the supabase repositories index.
7. **`packages/integrations-drivers` package** — AWS + GCP drivers, registry, unit tests with SDK/fetch mocks.

### Stage C — server

8. **Server routes** — `apps/server/src/routes/integrations.ts`, mount point, route tests. Includes rate limiter, redaction list update, driver registry bootstrap.
9. **`ISecretVault.forget` implementation** — whichever concrete vault is wired in `repositories-factory.ts` (§7.7). If not supported, log once on server boot and leave `forget` undefined.

### Stage D — web wiring

10. **HTTP adapter** — `apps/web/src/lib/repositories/integration-connection.repository.ts`. Wire into `repositories-factory.ts`.
11. **Shell runtime resource** — `packages/shell-runtime/src/resources/integrations.ts`, export on `ShellClient`.
12. **`packages/apps/integrations` plugin app** — §7.9. Imports the already-validated components from `@qlm/features-integrations`; adds only the `useShell()` + React Query wiring + cache invalidation + toasts. Because the components have been visually validated in stage A, this step should reduce to glue.

### Stage E — polish & verification

13. **i18n keys across all locales** — §11 map into `packages/i18n/src/locales/*/integrations.json`. (The feature package has been using these keys all along from stage A; this step just ensures the non-English locales are populated.)
14. **Feature flag** (optional) — `VITE_FEATURE_INTEGRATIONS`, default off in prod.
15. **Manual smoke** — §10.5 checklist against `pnpm dev` with a real sandbox AWS account and a real sandbox GCP project.
16. **Playwright smoke tests** — §10.4, follow-up if Playwright credentials are not wired yet.

**Hard rules for this sequencing**

- No work in stage B begins until the stage-A review gate is passed.
- Nothing in `packages/features/integrations` ever imports from `@qlm/shell-runtime`, `@tanstack/react-query`, or any repository package. If such an import becomes tempting, the data flow has slipped into the wrong layer — move it up to `packages/apps/integrations`.
- Stories in stage A are written against the *same* fixtures the component tests use, so a props-shape change in stage B fails both the story and the test at once and nothing drifts.

## 13. Follow-ups (not in this spec, tracked for phase 2+)

- Replace in-memory rate limiter with a distributed one (Redis or Supabase row-based) when `apps/server` scales horizontally.
- Promote reads from HTTP to direct-Supabase once there is pressure to reduce latency on the list view.
- Add `integrations.view` as a finer-grained permission if some orgs want to hide integrations entirely from non-admins.
- Persist a short `test_history` (last N results) if ops tells us one-level status is not enough to debug outages.
- Surface region capabilities (e.g. "RDS available") once phase 2's discovery code lands — currently the regions panel only shows ids.
